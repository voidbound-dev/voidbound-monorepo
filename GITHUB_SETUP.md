# Инструкция по настройке инфраструктуры GitHub

Для эффективной разработки проекта "Voidbound: Chronoscape" рекомендуется использовать структуру монорепозитория или связанных репозиториев внутри GitHub Organization.

## 1. Создание Организации
Создайте организацию на GitHub (например, `voidbound-dev`), чтобы объединить все ресурсы проекта.

## 2. Структура Репозиториев

Рекомендуется создать один основной монорепозиторий для кода и несколько вспомогательных:

1.  **`voidbound-monorepo`** (Основной)
    -   `apps/client`: Frontend (Babylon.js + React).
    -   `apps/server`: Backend сервисы.
    -   `packages/shared`: Общие типы, DTO, константы.
    -   `packages/domain`: Ядро бизнес-логики (DDD).
2.  **`voidbound-assets`** (LFS)
    -   Хранение тяжелых ассетов: 3D модели, текстуры, звуки. Используйте Git LFS.
3.  **`voidbound-docs`** (Опционально)
    -   Развернутая вики проекта, дизайн-документы, арты.

## 3. Настройка GitHub Projects (Управление задачами)

1.  В организации перейдите в раздел **Projects** -> **New Project** -> **Board**.
2.  Назовите проект: `Voidbound: Chronoscape Roadmap`.
3.  Создайте следующие колонки (Status):
    -   `Backlog`: Все идеи и будущие задачи.
    -   `Ready for Dev`: Задачи, прошедшие декомпозицию.
    -   `In Progress`: Текущие задачи.
    -   `In Review`: Задачи на проверке (PR open).
    -   `Done`: Завершенные задачи.
4.  Используйте **Milestones** для отслеживания этапов роадмапа (например, `M1: Prototype`).

## 4. GitHub Actions (CI/CD)

Настройте в основном репозитории `.github/workflows/`:
-   `ci.yml`: Запуск тестов (TDD!), линтинг и сборка при каждом PR.
-   `deploy.yml`: Автоматический деплой на staging/production (например, через Vercel для фронтенда и Docker для бэкенда).

## 5. Branch Protection Rules

Настройте правила для ветки `main`:
-   Require a pull request before merging.
-   Require status checks to pass before merging (CI tests).
-   Require conversation resolution before merging.

## 6. Первые шаги в репозитории

1.  Инициализируйте монорепозиторий (например, с помощью `turborepo` или `nx`).
2.  Добавьте `README.md`, `ROADMAP.md` и `.junie/guidelines.md` в корень.
3.  Настройте GitHub Issues с шаблонами.
    -   В репозитории уже созданы файлы в директории `.github/ISSUE_TEMPLATE/`.
    -   GitHub автоматически распознает их и предложит при создании нового Issue.
    -   Для настройки через UI: **Settings** -> **Features** -> **Issues** -> **Set up templates**.

## 7. Детальная настройка GitHub Issues

Для поддержания порядка в разработке мы используем структурированные шаблоны.

### Как это работает:
1.  **Bug Report**: Используется для сообщения о багах. Содержит шаги воспроизведения и ожидаемое поведение.
2.  **Feature Request**: Используется для предложения новых механик (например, новые Skill Orbs или изменения в Atlas).

### Шаги по ручной проверке/настройке:
1.  Перейдите во вкладку **Settings** вашего репозитория.
2.  В разделе **General** найдите подраздел **Issues**.
3.  Нажмите кнопку **Set up templates**.
4.  Вы можете отредактировать существующие `Bug Report` и `Feature Request` прямо в браузере.
5.  Рекомендуется также включить **Issue Forms** (YAML), если проект станет публичным и потребуется более строгая валидация полей.
